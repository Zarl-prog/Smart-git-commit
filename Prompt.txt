PART 4 — THE UNIQUE COMMIT MESSAGE FORMAT (most important part)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

This is what makes this skill different from every other agent. The commit
message format must be BOTH machine-readable AND deeply human-readable.

Every commit message follows this exact 5-part structure:

────────────────────────────────────────────────
<type>(<scope>): <imperative summary — max 72 chars>

CONTEXT: <1 sentence — what state the code was in BEFORE this change>
CHANGE:  <1 sentence — exactly what was done>
WHY:     <1-2 sentences — the reason, business or technical motivation>
IMPACT:  <1 sentence — what this enables or unblocks, or "No breaking changes">

<footers: Closes #N | BREAKING CHANGE: ... | Perf: before→after>
────────────────────────────────────────────────

Rules for each field:

CONTEXT — Describes the problem or gap that existed. Past tense.
  Good:  "Auth tokens had no expiry, allowing indefinitely valid sessions."
  Bad:   "There was a bug."

CHANGE — What the commit actually does. Present tense, specific.
  Good:  "Adds 15-minute sliding expiry with refresh rotation."
  Bad:   "Fixed the token thing."

WHY — Business or technical reason. Not obvious from the code.
  Good:  "Required for SOC2 compliance; also fixes session hijacking vector."
  Bad:   "To improve security."

IMPACT — What downstream effect this has. What can now be done?
  Good:  "Enables audit logging of token refresh events in Phase 2."
  Bad:   "Things are better now."

Build these rules into SKILL.md Phase 5 with inline examples.
Also build them into references/message-examples.md with 15 full examples
covering: feat, fix, perf, security, refactor, test, docs, chore, breaking
change, WIP, hotfix, revert, release, deps, and migration.

PART 5 — SCRIPTS (all must be production quality)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

scan-secrets.sh:
  - #!/bin/bash + set -e
  - Grep for: AWS keys, API tokens, private keys, passwords, .env values,
    Bearer tokens, connection strings, Stripe keys, GitHub tokens
  - Check for staged .env .pem .key credentials files
  - Color output: GREEN "✓ Clean" or RED "✗ Secret found: <file>:<line>"
  - Exit 0 = clean, exit 1 = found (caller must block commit)
  - stderr for progress, stdout for JSON result {status, findings[]}

detect-test-runner.sh:
  - Auto-detect: package.json→npm test, pytest.ini/pyproject.toml→pytest,
    Cargo.toml→cargo test, go.mod→go test ./..., Makefile→make test,
    .rspec→bundle exec rspec, mix.exs→mix test, build.gradle→./gradlew test
  - Run detected suite, capture output
  - stdout: JSON {runner, status:"pass"|"fail"|"not_found", test_count, failures[]}
  - If not_found: warn to stderr, exit 0 (don't block — warn only)

split-commits.sh:
  - Read git diff --stat
  - Group files by directory/module/type
  - Print proposed split plan as numbered list
  - Ask user to confirm (read -p)
  - Stage and commit each group in sequence
  - stdout: JSON {commits_made: N, messages: []}

generate-changelog.sh:
  - Read git log since last tag (or all history if no tags)
  - Group by type: Features, Bug Fixes, Performance, Security, Breaking Changes,
    Other
  - Extract CONTEXT/CHANGE/WHY from our commit format
  - Write Markdown section with version + date
  - Prepend to CHANGELOG.md (create if missing)
  - stdout: JSON {version, entries_added: N}

create-pr.sh:
  - Require gh CLI available (check first, error clearly if not)
  - Read branch name, last N commits, open issues
  - Build PR title from most significant commit subject
  - Build PR body from templates/pr-body.md filled with commit contexts
  - Create as draft by default (--draft flag)
  - Print PR URL to stdout
  - stdout: JSON {pr_url, pr_number, draft: true}
